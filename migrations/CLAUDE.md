# migrations/

Alembic, async template (`alembic init -t async`). `env.py` is wired to
`app.config.get_settings().database_url` and imports
`app.infrastructure.repositories` so `Base.metadata` sees every registered
model — don't hand-edit the DB URL here.

## Workflow

1. Add/change a model in `app/infrastructure/repositories/<concept>.py`
   (and make sure it's imported by `repositories/__init__.py`).
2. `uv run alembic revision --autogenerate -m "<message>"`.
3. **Read the generated file.** Autogenerate is a starting point, not
   truth — check `upgrade()`/`downgrade()` match your intent, delete any
   spurious/unrelated ops it picked up, and confirm portable types (`Uuid`,
   `DateTime(timezone=True)`, not SQLite-specific ones).
4. Apply it to a throwaway DB and inspect the schema before committing
   (`make migrate` applies to the real one), then `uv run alembic check`
   to confirm no drift between models and migrations.

Ruff still lints this directory — autogenerated files aren't exempt
(`pyproject.toml`'s `[tool.mypy]` excludes `migrations/`, but
`[tool.ruff]` does not). Run `uv run ruff check migrations/` after
autogenerating and fix violations (usually `UP007`/`UP035`/import sorting)
before committing — this bit us once already (a migration shipped with 5
lint violations that went unnoticed for several tasks).

Autogenerate can also render a custom column type (a `TypeDecorator`
subclass, e.g. `_UTCDateTime` in `app/infrastructure/repositories/events.py`)
as an unimported dotted Python path — a runtime `NameError` on upgrade, not
just a lint nit. Fix it by replacing that reference with the type's actual
DDL-level `impl` (e.g. `sa.DateTime(timezone=True)`): a `TypeDecorator` only
changes Python-side value processing, not the DDL, so the swap is safe.
